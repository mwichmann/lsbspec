Decisions Made

Because this task-force is focused on building working implementations quickly many of the decisions on what to make part of the "standard" are being made purely on pragmatic considerations. If an aspect of what would be needed as part of the system already has a working implementation that is widely available, it may just become part of the "standard" as it is to speed the building of reference implementations.

What file format will the binary packages use?

A specific version of the RPM binary package format. This is expected to be a version compatible with both the current RPM v3 and v4 package manager implementations currently in existence.

Why the RPM file format?

Pragmatism. The RPM binary package file format is a proven, robust, backward and forward compatible format. It's currently used by many distributions and is understood boy others.

What file format will the source packages use?

An ad-hoc gzip compressed GNU tarball with an lsb-pkg-construct shell (sh) script in its root. Details to be determined.

What is the base software package in the trivial package example?

The trivial package example is intended to be just enough to test out the packaging system from end to end from building to delivery to installation.

The GNU example "Hello World" software project should be enough :). It has executable and documentation components and some dependence on LSB mandated libraries.

What is the trivial package example?

A simple "Hello World" example would be enough to test out the packaging system from end to end. The GNU example "Hello World" software project should be enough.

What is the "reasonably complex" package example?

GNU "Hello World" :).

Seriously however, we need a chunk of software that is:

  • delivered in several parts: some needed by the others and others optional.

  • of manageable size.

  • easy to test from a packaging standpoint.

  • already works on a number of distributions.
